# Point Decision - Resolve Cross-rhei dependency readiness

- Chosen solution: S-001 as the normative readiness rule, with S-003-style implementation latitude: a cross-rhei dependency unblocks dependent work only when the prior task satisfies the same successful-terminal rule used by normal scheduling, meaning the prior's resolved state has `final: true` and its normalized state name is not `cancelled`.
- Why chosen: This directly satisfies all stated constraints and both discussions converged on it. It preserves identical scheduling semantics across rhei boundaries while avoiding unnecessary deployment assumptions: implementations may use a shared callable predicate, direct remote state-machine resolution, or an exported readiness result, provided the result is computed from the prior rhei's own state-machine semantics and is fresh or otherwise auditable.
- Alternatives considered: S-002's single shared predicate is strong when local and cross-rhei scheduling run in the same codebase because it reduces drift and simplifies tests, but requiring one shared in-process callable over-constrains cross-process, cross-repository, or serialized/API boundaries. Direct synchronous dependent-side resolution of the prior rhei's state machine is transparent and sufficient, but it is not necessary if the prior rhei exports an equivalent predicate result with explicit state-machine version and freshness/invalidation guarantees. Treating any terminal state as ready was rejected because normalized `cancelled` must not unblock dependent work.
- Remaining uncertainty: The final implementation still needs to specify the freshness and invalidation contract for cached or exported readiness data. If the prior state, prior state-machine meaning, or exported readiness result cannot be resolved reliably, the dependency must remain blocked.
- Effect on final solution: The final answer must define cross-rhei readiness as the same successful-terminal predicate used by normal scheduling: `final: true` and normalized state not `cancelled`, evaluated with the prior rhei's state-machine semantics. It should prefer a shared predicate where practical, allow direct resolution or an exported/cached readiness result only when behaviorally equivalent and fresh/auditable, and require unresolved, stale, non-final, or normalized-`cancelled` prerequisites to keep dependents blocked.
