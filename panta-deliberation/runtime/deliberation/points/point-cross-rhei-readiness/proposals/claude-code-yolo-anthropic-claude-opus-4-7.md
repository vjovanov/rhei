# Proposal - claude-code[yolo]:anthropic:claude-opus-4-7

- Recommendation: Define a single normative predicate `is_successful_terminal(prior)` that returns true iff the prior's resolved state has `final: true` and its normalized name is not `cancelled`, and require both local scheduling and cross-rhei readiness checks to call this same predicate against the prior task's state as resolved by the prior's own rhei state machine. Cross-rhei readiness is thus defined by composition (resolve remote state with the remote machine, then apply the shared predicate), not by a separate rule.
- Reasons: Expressing readiness as one shared predicate makes "same rule as normal scheduling" enforceable rather than aspirational — there is one definition to update, one to test, and one to cite from §-marked code. Resolving the prior's state with the prior's own state machine handles the cross-rhei aspect correctly: each rhei may declare its own state machine, and only that machine knows which of its states are `final` and which normalize to `cancelled`. A non-terminal, non-final, or normalized-cancelled prior all collapse to "not ready" uniformly, which matches the constraint that cancellation must not unblock dependents.
- Tradeoffs: The implementation must reach into the prior's rhei to read its state machine, which adds a cross-rhei lookup on every readiness check (cacheable, but a new dependency). It treats `cancelled` as the *only* unsuccessful terminal — if future state machines introduce other unsuccessful terminals (e.g. `abandoned`, `rejected`), they must either normalize to `cancelled` or this predicate must be extended; that extension point should be a separate decision so it does not silently broaden here. Cancelled priors leave dependents permanently blocked without an explicit override path, which is correct by the constraint but may surprise users — the UX cost is visible blockage rather than silent unblocking.
- Assumptions: Each rhei's state machine is accessible to the readiness checker (either co-resident in the Panta or fetchable through the same project root). The `final: true` flag and the normalization to `cancelled` are properties of state definitions, not of individual tasks, and are stable for a given state machine version. The prior's current state is observable and unambiguous (one current state per task). Normalization of state names to `cancelled` is already defined elsewhere in the spec and produces a single canonical form.
- Rejection criteria: Do not use this proposal if cross-rhei readiness must intentionally diverge from local scheduling (e.g. allowing dependents to proceed past a cancelled cross-rhei prior by policy), if multiple unsuccessful terminal states beyond normalized `cancelled` must block dependents in this change (extending the predicate inline would exceed this point's scope), or if the prior's rhei state machine cannot be resolved synchronously at readiness-check time and an async/eventual model is required instead.
