# Discussion Response - codex[yolo]:openai:gpt-5.5

- Changed position: yes. I still support S-001, but my earlier S-003 wording was too loose around "equivalent normalized readiness result." I now support making `is_successful_terminal(prior)` the normative readiness predicate, with one caveat: cross-rhei boundaries do not have to call the same in-process function if the prior rhei exports a result that is explicitly produced from the same predicate and carries enough state/version/freshness evidence to audit that fact.
- Remaining disagreements: D-001 only as an implementation-boundary distinction: one normative predicate should be required, but not necessarily one shared callable across process, repository, or serialized boundaries. D-002 only in who performs the resolution: the prior rhei's state-machine semantics must be authoritative, whether the dependent side reads them directly or consumes the prior rhei's exported predicate result. D-003 should be resolved by contract: synchronous lookup is sufficient but not mandatory; cached or exported readiness is acceptable only with explicit freshness/invalidation guarantees, and unavailable or stale readiness must be treated as not ready.
- Concessions: claude-code is right that a single normative predicate is the cleanest way to make "same readiness rule as normal scheduling" enforceable instead of aspirational. It is also right that state meaning must come from the prior rhei's own state machine, because `final: true` and normalized `cancelled` are properties of that rhei's state definitions, not names the dependent side can safely guess.
- Final recommendation for this point: adopt S-001 as the user-visible rule and S-002 as the normative implementation contract: a cross-rhei dependency unblocks only if the prior task satisfies `is_successful_terminal(prior)`, defined as the prior's resolved state having `final: true` and the normalized state name not being `cancelled`. The predicate must be evaluated using the prior rhei's state-machine semantics. Direct dependent-side resolution is acceptable; an exported/cached readiness result is also acceptable only if it is defined as the prior rhei's application of that predicate and includes freshness/version guarantees. If the prior state, state-machine meaning, or exported result cannot be resolved reliably, the dependency remains blocked.
- Reason: This handles D-001 by preventing semantic drift without over-specifying deployment topology. It handles D-002 by making the prior rhei authoritative while still allowing a lower-coupling readiness API. It handles D-003 by preserving correctness under synchronous, cached, or exported evaluation: only fresh, auditable successful-terminal evidence can unblock, and cancelled or uncertain prerequisites cannot.
