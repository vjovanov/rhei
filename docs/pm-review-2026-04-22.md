A hands-on run of the `rhei` CLI exercising the main user-visible workflows: `templates`/`instantiate`, `validate`, `render`, `next`, `transition`, `complete`, `reset`, `run --dry-run`, and `install-skills`. ## High-severity issues (affect correctness or core UX)

### 1. `rhei next` error "no tasks are ready to claim" conflates three distinct situations

Repro:
```
### Task 1: Write
**State:** pending
```
```
$ rhei next plan.rhei.md
أ— no tasks are ready to claim
```

The task is not blocked â€” it is in `pending`, which has four outgoing transitions (`agent-review`, `human-review`, `completed`, `cancelled`). `rhei next` cannot auto-pick because the choice requires human/agent judgment, but the error tells the user the opposite ("no tasks are ready"). A user can't tell from this message whether:
- (a) every task is in a terminal state (nothing left to do),
- (b) every ready task has ambiguous outgoing transitions (use `rhei transition`), or
- (c) every ready task is blocked by prerequisites.

**Fix direction:** When `next` finds eligible tasks but can't decide, say so explicitly and list the candidate transitions plus the `rhei transition â€¦` command line to pick one. Reserve "no tasks are ready" for the genuine no-candidates case.

### 2. `rhei reset` on a plain-file plan leaves stale `> **Result:**` links and result files

Repro: complete a task with `rhei complete`, then `rhei reset plan.rhei.md`.

```
### Task 2: Review draft
**State:** draft                       â†گ reset succeeded
**Prior:** Task 1

> **Result:** [2](runtime/results/2.md)   â†گ NOT removed
```

And `runtime/results/2.md` still exists on disk. For workspaces the code does clean up `runtime/` (`reset_command` at `crates/rhei-cli/src/main.rs:7087`), but for plain plans it doesn't. A user who runs `reset` reasonably expects to be back at a clean slate; instead they now have dangling content inside task bodies that a fresh `complete` will duplicate.

**Fix direction:** either (a) have `reset` on plain plans also strip `> **Result:**` lines and delete `runtime/results/*.md`, or (b) document loudly that reset only affects state.

### 3. `rhei transition` checks transition legality before state mismatch

Repro: a task is in `draft`, user types `--from pending --to pending`:
```
أ— transition from 'pending' to 'pending' is not allowed by the state machine
```

The real problem is the user's `--from` is wrong (the task is in `draft`, not `pending`). The legality-first ordering gives a cryptic answer. With `--from pending --to completed` the error is correctly "conflict: Task 1 is in state 'draft', expected 'pending'". The ordering should always prefer the state-mismatch error; that's the one the user can act on.

**Fix direction:** check current-state vs `--from` first; only fall through to the legality check when they match.

### 4. `rhei complete` on a non-terminal loop state gives an unactionable error

Repro (`spec-review` workspace mid-loop):
```
$ rhei complete â€¦ --task spec-review --result "Done"
أ— transition from 'fix' to 'completed' is blocked by loop budget or condition
```

"Loop budget or condition" is developer-vocabulary. A user/agent receiving this has no signal about what to do next (advance the visit count? finish the current cycle? override?). The error should name the specific condition (`visitCount < visits` with `visitCount=1, visits=2`) and point at the transition(s) that *are* currently available from `fix`.

### 5. ANSI color leaks through pipes in `render --format progress`

```
$ rhei render plan.rhei.md --format progress | cat
* Task 1: Write initial draft  ^[[35m[DRAFT]^[[0m
```

No TTY detection, no `NO_COLOR` environment-variable support. `--no-color` exists but is opt-in. This is the industry convention violation most likely to cause downstream tooling to break (logs with embedded escape codes are near-unreadable).

**Fix direction:** default color to auto (on when stdout is a TTY, off otherwise), respect `NO_COLOR`, keep `--no-color` as an explicit override.

### 6. Parser bails on the first error

```
### Task 1: Missing state          â†گ no **State:**
### Task 2: Typo
**State** pending                  â†گ missing colon
```
Only Task 1's error is reported. Users with a batch of authoring problems have to fix-and-retry repeatedly. For plan files with a dozen tasks this is a meaningful productivity tax â€” the `miette` framing is polished, but the content surface is one error at a time.

**Fix direction:** accumulate multiple parse errors per run. The existing validator already does this (cycles + missing deps + state-check in one pass); the parser should match.

### 7. `instantiate` does not validate `type: path` inputs

```
$ rhei instantiate spec-review --set spec=/nonexistent/file.md --output /tmp/bad
Validation succeeded
Instantiated template 'spec-review' into '/tmp/bad'.
```

The `spec` input is declared `type: path` (see `.agents/rhei/templates/spec-review/template.yaml`). A user would reasonably expect `path` to mean "must refer to an existing filesystem path." Currently it is indistinguishable from `type: string`. The typo surfaces late â€” typically when an agent tries to read the file midway through execution, leaving a half-instantiated workspace behind.

**Fix direction:** have `type: path` check existence at instantiation; add `type: file` / `type: directory` if finer-grained is wanted; or rename to `type: string` if validation is out of scope.

### 8. `--json` output format does not apply to errors

```
$ rhei next /does/not/exist --json
أ— failed to read input file '/does/not/exist': No such file or directory
```

A machine consumer that opts into JSON has to parse two output shapes (JSON on stdout, miette text on stderr). The `next` success path is structured; the error path isn't.

**Fix direction:** when any `--json` flag is present, emit errors as `{ "error": { "message": â€¦, "kind": â€¦, "path": â€¦ } }` to stderr.

## Medium-severity issues (friction, not blocking)

### 9. `rhei transition` silently creates a result file with no body

After `rhei transition â€¦ --from pending --to completed` with no `--result` argument:
```
$ cat runtime/results/1.md
## pending â†’ completed

```
The file is created but contains only a header; it is not linked from the task body (only `rhei complete` adds the `> **Result:** â€¦` link). The user ends up with a workspace where some tasks have linked results and others have orphaned `.md` files in `runtime/results/`. This is confusing and surprising â€” users expect the result file to be an artifact of `complete`, not `transition`.

**Fix direction:** pick one â€” either `transition` should not touch `runtime/results/`, or it should also insert the link. Current halfway state is the worst of both.

### 10. `rhei templates` from outside a project silently shows nothing

```
$ cd /tmp && rhei templates
No templates found.
```
The message doesn't say *where* it looked (project `./.agents/rhei/templates`, user `~/.config/â€¦`, etc.). A user who just installed `rhei` has no hint that templates are per-project and they're in the wrong cwd.

**Fix direction:** list the search paths that were checked and found empty, e.g. `No templates found in: ./, ~/.config/rhei/templates, â€¦`.

### 11. Un-interpolated template variables leak into human-visible output

State-machine `instructions` in `.agents/rhei/templates/spec-review/states.yaml` contain literal `{visit_count} < {visits}` strings. `rhei next` prints them verbatim:
```
â€¦immediately transition back to `review` when `1 < 2`;
otherwise transition to `completed`.
```
`{task_id}` and `{visit_count}` *are* interpolated (visible in the output above), but the boolean-looking `{visit_count} < {visits}` is meant as human-readable explanation, not evaluated. A reader can't easily tell which of these are live variables versus instructional pseudocode. Worth a convention in the docs, or a distinct delimiter for live substitution.

### 12. `Agent: codex (default)` is ambiguous formatting

Printed in the human-mode `next` output, the `(default)` parenthetical reads like "codex is the default agent." In fact it means `model=default`, with the agent being explicitly `codex` (set by the state machine). Rewriting as `Agent: codex | Model: default` â€” or better, showing the two on separate lines â€” removes the ambiguity.

### 13. JSON `metadata.metadata.tasks.â€¦` double-nesting

`rhei render --format json` on a workspace produces:
```json
"metadata": {
  "metadata": {
    "tasks": { "spec-review": { "stateVisits": { "review": 1 } } }
  }
}
```
The duplicated `metadata` key is the serialized YAML frontmatter (which itself has a top-level `metadata:` key) wrapped in the renderer's own `metadata` field. Machine consumers have to know to double-dive. Either flatten, or rename the renderer's wrapper to `frontmatter` to make intent obvious.

### 14. `validate` source-phrasing reads as a placeholder

```
I validated this plan using states from (built-in default), but found a problem.
```
`(built-in default)` in parentheses after "from" scans like unreplaced template text. "using the built-in default state machine" reads cleanly.

## Known gaps (called out so they don't get lost)

### 15. `**Assignee:**` is half-implemented

After the PR from 2026-04-22:
- Parser extracts `**Assignee:**` and it round-trips through JSON/markdown output.
- `rhei complete` still strips the line (works as before).
- **Missing:** `rhei next` does not write `**Assignee:** <agent>` on claim, despite `docs/specs/rhei-next.spec.md` lines 21â€“43 calling for it.
- **Missing:** the claim-blocking rule ("a task is not claimable if `**Assignee:**` is already set") is not enforced â€” `rhei next` advances a task whose assignee is already populated.

This was explicitly scoped out of the parse PR. It should be tracked as the immediate follow-up since the authoring shape is already spec'd and the parser now supports it.

### 16. `--peek` on `rhei next` is referenced in spec but absent from CLI

`docs/specs/rhei-next.spec.md` line 82 describes a `--peek` flag ("must not modify the field"). `rhei next --help` does not list `--peek`. Either the spec is aspirational or the flag was removed â€” worth reconciling.

## What works well

- The `miette` error framing (with code frames and file context) is a clear step above raw compiler-style messages. Keep investing there.
- `rhei states` is a genuinely useful introspection tool â€” one command to see the whole state machine, transitions, and per-state instructions/artifacts.
- `rhei validate` catches cycles, self-dependencies, missing prerequisites, and missing state lines with clear messages. The validator is noticeably more mature than the parser on the "report multiple problems" axis.
- `rhei instantiate --list-inputs` is a nice UX for users discovering a template. Keep.
- `install-skills --dry-run --local` is discoverable and readable â€” it shows exactly which files would land where, across a reasonable set of agent targets.
