# State Machines and Transitions Discrepancy Audit

Task: `state-machines-transitions`

Scope source: `runtime/spec-implementation-discrepancy-audit/state-machines-transitions/scope.md`

This audit compares the scoped state-machine and transition claims against implementation code, tests, shipped templates, examples, and CLI-observable behavior where practical. It records discrepancies only; it does not propose fixes.

## SMT-001: YAML State Machine Root Schema and Default Machine

- **implementation-diverges**: The specification says the machine-readable default is `docs/specs/states.yaml`, version `3.0`, with `profiles` and `node_policy`. The compiled default used by `StateMachine::builtin_default()` is `crates/rhei-validator/src/default-states.yaml`, version `2.0`, with `draft.initial: true` and no profiles/node policy. Evidence: `docs/specs/states.yaml:5`, `docs/specs/states.yaml:107`, `docs/specs/states.yaml:125`; `crates/rhei-validator/src/default-states.yaml:5`, `crates/rhei-validator/src/default-states.yaml:16`; `crates/rhei-validator/src/lib.rs:641`.
- **implementation-diverges**: State machines without `profiles` and `node_policy` still load as legacy machines, even though the current specs mark both as required and state-level `initial: true` forbidden. Evidence: `docs/specs/rhei-states.spec.md:91`; `docs/specs/rhei-states.spec.md:101`; `crates/rhei-validator/src/lib.rs:631`; `crates/rhei-validator/src/lib.rs:1021`.
- **missing-validation**: `transitions` is declared required by the transition spec, but `StateMachine` defaults it to an empty list when omitted. Evidence: `docs/specs/rhei-transitions.spec.md:536`; `crates/rhei-validator/src/lib.rs:626`.
- **no-discrepancy**: `rhei complete` selects a one-hop non-cancelled terminal transition and refuses to fall back to `cancelled`. Evidence: `crates/rhei-cli/src/main.rs:9615`; tests at `crates/rhei-cli/src/main.rs:11597`, `crates/rhei-cli/src/main.rs:11617`, and `crates/rhei-cli/src/main.rs:11634`.

## SMT-002: Profiles and Node Policy

- **implementation-diverges**: The spec defines `overrides[].match` as an object with optional `type` and `level`, but the implementation models it as a string `pattern` matched exactly against task id. Evidence: `docs/specs/rhei-states.spec.md:778`; `docs/specs/rhei-states.spec.md:791`; `crates/rhei-validator/src/lib.rs:601`; `crates/rhei-validator/src/lib.rs:672`.
- **missing-validation**: The spec requires each profile `allowed` set to contain at least one final state and every non-final state to have an allowed path to a final state. `validate_profiles_and_node_policy` validates names, membership, duplicates, and references, but does not enforce final-state presence or graph reachability. Evidence: `docs/specs/rhei-states.spec.md:740`; `crates/rhei-validator/src/lib.rs:1046`.
- **missing-validation**: The spec requires `by_type` keys and override `type` values to be declared non-root node kinds and override `level` values to fit the plan structure. State-machine loading validates only empty/reserved `by_type` keys and undefined profiles; override level/type are not represented. Evidence: `docs/specs/rhei-states.spec.md:812`; `docs/specs/rhei-states.spec.md:817`; `crates/rhei-validator/src/lib.rs:1113`; `crates/rhei-validator/src/lib.rs:1142`.
- **implementation-diverges**: `rhei reset` resets all task files to a single state-level `initial: true` value rather than each node's resolved profile `initial`. Evidence: `docs/specs/rhei-states.spec.md:802`; `crates/rhei-cli/src/main.rs:9395`; `crates/rhei-cli/src/main.rs:9446`; `crates/rhei-cli/src/main.rs:9570`.
- **implementation-diverges**: `rhei next` claimability and initial auto-transition logic still key off `StateDef::initial`, not resolved profile initial. Evidence: `crates/rhei-cli/src/main.rs:8715`; `crates/rhei-cli/src/main.rs:9175`.

## SMT-003: Per-State Fields and Validation

- **missing-validation**: State `description` is required by the transition and writer specs, but `StateDef.description` is optional and explicitly permissive. Evidence: `docs/specs/rhei-transitions.spec.md:571`; `docs/specs/rhei-state-machine-writer.spec.md:70`; `crates/rhei-validator/src/lib.rs:478`.
- **missing-validation**: Target selectors are parsed for grammar, but the validator does not resolve target agent ids or modes against the merged agent registry at validation time as required by the states spec. Evidence: `docs/specs/rhei-states.spec.md:112`; `docs/specs/rhei-states.spec.md:114`; `crates/rhei-validator/src/lib.rs:749`; `crates/rhei-validator/src/lib.rs:394`.
- **ambiguous-spec**: The states spec says `agent` on a gating state is a validation warning, while program/tooling on gating states are validation errors. The implementation has no visible state-machine warning channel for this case, and no check appears in `validate_model_configuration` beyond final-state agent rejection. Evidence: `docs/specs/rhei-states.spec.md:126`; `crates/rhei-validator/src/lib.rs:787`.
- **no-discrepancy**: Several structural validations are implemented: `target`/`all_targets` mutual exclusion and legacy-field exclusion, `model`/`all_models` checks, visit `0` rejection, agent/program mutual exclusion, program shape and final/gating exclusions, timeout parsing, artifact name/path validation, poll shape validation, and MCP/skill duplicate/exclusion checks. Evidence: `crates/rhei-validator/src/lib.rs:732`; `crates/rhei-validator/src/lib.rs:772`; `crates/rhei-validator/src/lib.rs:778`; `crates/rhei-validator/src/lib.rs:828`; `crates/rhei-validator/src/lib.rs:883`; `crates/rhei-validator/src/lib.rs:923`; `crates/rhei-validator/src/lib.rs:1258`; `crates/rhei-validator/src/lib.rs:1322`; `crates/rhei-validator/src/lib.rs:1556`.

## SMT-004: Gating and Terminal Behavior

- **implementation-diverges**: Manual transition execution does not reject transitions from final states before applying exact or wildcard rules. The transition spec says final states are absorbing and wildcard transitions do not match final states. Evidence: `docs/specs/rhei-transitions.spec.md:766`; `crates/rhei-cli/src/main.rs:5060`.
- **implementation-diverges**: `rhei complete` does not explicitly reject a gating current state. It rejects terminal states and children, then searches for a terminal transition; `find_completion_state` has no gating check. Evidence: `docs/specs/rhei-states.spec.md:73`; `crates/rhei-cli/src/main.rs:9322`; `crates/rhei-cli/src/main.rs:9340`; `crates/rhei-cli/src/main.rs:9619`.
- **no-discrepancy**: Ready-task discovery skips terminal and gating states, so `rhei run`/callback scheduling and automatic ready selection avoid autonomous work from gating states. Evidence: `crates/rhei-cli/src/main.rs:8670`; `crates/rhei-cli/src/main.rs:8692`; `crates/rhei-cli/src/main.rs:8437`.

## SMT-005: Counted Visits and Counted Loops

- **no-discrepancy**: Basic counted-loop mechanics are implemented: `visits: 0` is rejected; `-1`, non-counted suffixes, and suffixes over budget are invalid; transition metadata stores `metadata.tasks.<id>.stateVisits.<state>`; state rendering writes suffixes only for visits greater than `1`; and `visitCount`/`visits` are condition operands. Evidence: `crates/rhei-validator/src/lib.rs:778`; `crates/rhei-validator/src/lib.rs:1887`; `crates/rhei-cli/src/main.rs:3788`; `crates/rhei-cli/src/main.rs:3850`; `crates/rhei-cli/src/main.rs:4112`; `crates/rhei-cli/src/main.rs:4136`.
- **implementation-diverges**: The spec says visit accounting is scoped per model or target when combined with `all_models` or `all_targets`. The implementation keys counters only by task id and state name, with no model/target dimension. Evidence: `docs/specs/rhei-transitions.spec.md:671`; `docs/specs/rhei-transitions.spec.md:672`; `crates/rhei-cli/src/main.rs:4136`; `crates/rhei-cli/src/main.rs:4148`.
- **missing-test**: The scoped tests cover counted loops generally, but I did not find a test that proves per-target or per-model visit accounting isolation. Evidence: tests named in `scope.md`; implementation key shape above.

## SMT-006: Polling States

- **no-discrepancy**: State-machine loading validates poll shape, duration, max attempts, final/gating exclusions, `visits` mutual exclusion, and self-loop presence. Evidence: `docs/specs/rhei-states.spec.md:145`; `crates/rhei-validator/src/lib.rs:923`; tests near `crates/rhei-validator/src/lib.rs:3856`.
- **implementation-diverges**: Poll runtime semantics are not wired into the CLI engine. There are no CLI references to `pollNextAttemptAt`, `pollAttempts`, or `pollMaxAttempts`; condition operands support only `visitCount`, `visit_count`, `visits`, and numeric task metadata. Evidence: `docs/specs/rhei-states.spec.md:183`; `docs/specs/rhei-states.spec.md:196`; `crates/rhei-cli/src/main.rs:3838`; `crates/rhei-cli/src/main.rs:3850`; command search found `pollAttempts` only in `examples/ci-heal/states.yaml:107`.
- **implementation-diverges**: Poll attempt caps are not used by loop-budget code because `state_visit_limit` only reads `StateDef::visits`, while poll states are required to omit `visits`. Evidence: `docs/specs/rhei-states.spec.md:185`; `docs/specs/rhei-states.spec.md:188`; `crates/rhei-cli/src/main.rs:3811`; `crates/rhei-validator/src/lib.rs:948`.
- **spec-stale**: The shipped `ci-heal` example documents the poll behavior but explicitly says it applies "Once the `poll:` block is wired through `rhei run`." Evidence: `examples/ci-heal/index.rhei.md:75`; `examples/ci-heal/states.yaml:17`.

## SMT-007: Artifact Contracts

- **no-discrepancy**: Artifact definitions enforce non-empty unique names, non-empty relative paths, no raw `..` escape, and output artifacts cannot be optional. Required inputs are skipped when optional and required outputs are checked. Evidence: `crates/rhei-validator/src/lib.rs:1556`; `crates/rhei-cli/src/main.rs:4842`; `crates/rhei-cli/src/main.rs:4882`.
- **implementation-diverges**: The transition spec says source outputs are checked after `on_leave`/`on_enter` callbacks complete but before the state write is committed. Implementation checks source outputs and target inputs before the state write and before `on_enter`; `on_enter` runs after the write, with rollback on failure. Evidence: `docs/specs/rhei-transitions.spec.md:750`; `crates/rhei-cli/src/main.rs:5263`; `crates/rhei-cli/src/main.rs:5283`; `crates/rhei-cli/src/main.rs:5309`; `crates/rhei-cli/src/main.rs:5335`.
- **no-discrepancy**: Optional input path and existence are exposed to instructions and programs. Evidence: `docs/specs/rhei-states.spec.md:251`; `crates/rhei-cli/src/main.rs:4598`; `crates/rhei-cli/src/main.rs:7082`.
- **implementation-diverges**: Some shipped example artifact paths use variables outside the current spec/runtime namespace (`{task.id}`, `{visit}`), so the resolver leaves them literal. Evidence: `docs/specs/rhei-states.spec.md:229`; `crates/rhei-cli/src/main.rs:4470`; `examples/ci-heal/states.yaml:43`; `examples/ci-heal/states.yaml:65`.

## SMT-011: Callback Declaration, Context, Result, and Error Semantics

- **no-discrepancy**: CLI callbacks support optional `on_leave`/`on_enter`, rejection, redirect, `data` accumulation, `success: false` with `nextState` downgrade, `--no-callbacks`, and `on_enter` rollback. Evidence: `docs/specs/rhei-transitions.spec.md:5`; `docs/specs/rhei-transitions.spec.md:775`; `crates/rhei-core/src/callback.rs:149`; `crates/rhei-napi/src/lib.rs:1`.

## SMT-013: Concurrent State Scheduling Metadata

- **no-discrepancy**: `StateDef.concurrent` exists and `rhei run` applies the non-concurrent scheduling rule to agent tasks, keeping fanout invocations for the same task together and deferring other tasks in the same non-concurrent state to later passes. Evidence: `docs/specs/rhei-states.spec.md:74`; `crates/rhei-validator/src/lib.rs:496`; `crates/rhei-cli/src/main.rs:7842`.
- **ambiguous-spec**: The spec says `concurrent` applies to `rhei run` scheduling generally, but the current implementation only applies the grouping/defer logic to the `agent_tasks` batch. Program and callback-only progression are sequential for other reasons, so there is no equivalent `concurrent` scheduling branch for them. Evidence: `docs/specs/rhei-states.spec.md:74`; `crates/rhei-cli/src/main.rs:7734`; `crates/rhei-cli/src/main.rs:8437`.
- **implementation-diverges**: Concurrent poll-state semantics depend on poll deadlines, which are not implemented in runtime scheduling. Evidence: `docs/specs/rhei-states.spec.md:211`; `crates/rhei-cli/src/main.rs:8674`.
