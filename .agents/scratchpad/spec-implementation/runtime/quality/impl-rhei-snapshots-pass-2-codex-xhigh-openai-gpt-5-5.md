# Quality Review pass 2 - target codex[xhigh]:openai:gpt-5.5

- Q-stationary-exits-skip-emit: Successful subprocess exits that leave the task in the same state can skip both auto and named snapshot emission.
  - Severity: high
  - File: crates/rhei-cli/src/cli/run_agent_mode.rs
  - Detail: Snapshot emission on the success path is only wired through state advancement or auto-transition selection. If an agent exits 0 but required outputs are missing, the code classifies the snapshot completion as `failure` but continues before emitting anything. If outputs are present but no outgoing transition is selected, `try_auto_advance_task` returns `Ok(None)` and the `Ok(None)` branch emits warnings without calling `emit_snapshots_after_agent_exit`. Both cases violate the requirement that every agent-state subprocess exit writes the auto `_state` snapshot, and they also suppress `snapshot.emit.on: failure`/`always` for the missing-output case.
  - Evidence: `snapshot_completion` becomes `Failure` when `outputs_ok` is false at `crates/rhei-cli/src/cli/run_agent_mode.rs:919-925`, but the missing-output branch continues without emission at `crates/rhei-cli/src/cli/run_agent_mode.rs:981-990`. The no-transition success branch handles `Ok(None)` without emission at `crates/rhei-cli/src/cli/run_agent_mode.rs:1029-1057`; the parallel path has the same shape at `crates/rhei-cli/src/cli/run_agent_mode.rs:1442-1552`. The spec says emit runs after every agent-state subprocess exit and writes auto snapshots regardless of completion classification (`docs/functional-spec/rhei-snapshots.spec.md:826-837`).
  - Suggested fix: Decouple success-path emission from transition application. After completion classification and transition selection, call `emit_snapshots_after_agent_exit` even when no transition is selected, except for the explicit skipped cases such as poll self-loops and non-agent states.

- Q-timeout-failure-emit-before-routing: Failure and timeout snapshots are emitted before selecting error/timeout routing.
  - Severity: medium
  - File: crates/rhei-cli/src/cli/run_agent_mode.rs
  - Detail: For non-zero exits and timeouts, the run loop emits snapshots immediately with `selected_to_state: None`, then later evaluates timeout routing. This bypasses the spec-mandated ordering where the orchestrator first selects the outgoing transition, including error or timeout routing and poll self-loop/exhaustion behavior, and only then emits snapshots before applying that transition. As a result, timeout/error self-loops cannot suppress poll-attempt emission, and emitted snapshots are not tied to the selected terminal route.
  - Evidence: The sequential path calls `emit_snapshots_after_agent_exit(..., None, ...)` before `fire_timeout_transition` at `crates/rhei-cli/src/cli/run_agent_mode.rs:926-945` and `crates/rhei-cli/src/cli/run_agent_mode.rs:1065-1077`. The parallel path does the same at `crates/rhei-cli/src/cli/run_agent_mode.rs:1387-1407` before `crates/rhei-cli/src/cli/run_agent_mode.rs:1553-1565`. The spec requires transition selection, including error/timeout routing and poll behavior, before emit decisions (`docs/functional-spec/rhei-snapshots.spec.md:828-832`, `docs/functional-spec/rhei-snapshots.spec.md:856-860`).
  - Suggested fix: Route failure and timeout outcomes through the same transition-selection hook used for successful auto-advance, pass the selected destination into `emit_snapshots_after_agent_exit`, and suppress emission when the selected poll transition is a self-loop.

- Q-pi-observed-model-not-parsed: Snapshot manifests never parse Pi session headers for observed provider/model.
  - Severity: medium
  - File: crates/rhei-cli/src/cli/snapshot_records.rs
  - Detail: The manifest writer always sets `observed_provider` and `observed_model` from the resolved/declared target. For Pi, the spec requires reading the session JSONL header at emit time because Pi may route to a provider/model different from the declared selector. This makes Pi manifests inaccurate and causes downstream cache-benefit diagnostics to compare inheritors against declared values rather than the actual observed runtime model.
  - Evidence: `write_snapshot_generation_atomic` writes `observed_provider` and `observed_model` directly from `resolved.model_provider` and `resolved.model_name` at `crates/rhei-cli/src/cli/snapshot_records.rs:506-507`, and the only cache-benefit check reads those manifest fields at `crates/rhei-cli/src/cli/snapshot_runtime_emit.rs:227-235`. The spec says Pi observed provider/model are read from the session JSONL header and only fall back to declared values with a warning when the header is absent or unparsable (`docs/functional-spec/rhei-snapshots.spec.md:364-366`, `docs/functional-spec/rhei-snapshots.spec.md:737-741`).
  - Suggested fix: Add a Pi-specific header parser before manifest construction, pass observed provider/model into `write_snapshot_generation_atomic`, and log a warning when parsing falls back to declared values.

- Q-invalid-session-layout-passes-validation: Static validation accepts snapshot session profiles that runtime cannot support.
  - Severity: medium
  - File: crates/rhei-cli/src/cli/settings_load_validate.rs
  - Detail: Validation treats any `session.layout` key as a supported layout and any non-empty `resume` object as preload support. Runtime later requires a parseable layout kind and extension, only locates FlatById transcript sources, and requires a concrete resume/fork flag. A statically resolved named `snapshot.emit:` or required `snapshot.inherit:` can therefore pass `rhei validate` and fail only at spawn time with `unsupported-snapshot-session`, contrary to the static author-contract rule.
  - Evidence: `profile_has_snapshot_layout` only checks `session.get("layout").is_some()` at `crates/rhei-cli/src/cli/settings_load_validate.rs:506-508`, and `profile_has_snapshot_resume` accepts any non-empty object at `crates/rhei-cli/src/cli/settings_load_validate.rs:514-523`. Runtime requires `snapshot_layout_manifest` to find `kind` and `ext` at `crates/rhei-cli/src/cli/snapshot_records.rs:182-185`, `transcript_source_for_snapshot` only supports `FlatById` at `crates/rhei-cli/src/cli/snapshot_records.rs:277-288`, and preload flag extraction requires a recognized `flag` at `crates/rhei-cli/src/cli/snapshot_records.rs:165-179`. The spec requires statically resolved named emits with no supported session layout to be validation errors (`docs/functional-spec/rhei-snapshots.spec.md:955-957`) and requires preload to have a supported session strategy (`docs/functional-spec/rhei-snapshots.spec.md:817-822`).
  - Suggested fix: Make settings validation use the same support predicates as runtime: validate recognized `SessionLayout` variants and required fields, and require a concrete resume or fork strategy flag for required preload.
