# Quality Review pass 1 - target codex[xhigh]:openai:gpt-5.5

- Q-targetless-auto-hard-error: Agent states without a resolved target now fail before spawn even when they do not declare snapshot operations.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/snapshot_runtime_2.rs
  - Detail: `preload_snapshot_inherit_before_spawn` calls `snapshot_target_slug_or_err(resolved)?` before it checks whether the current state declares `snapshot.inherit` or whether `--from-snapshot` is present. That makes snapshot target resolution mandatory for every agent-bearing state. The spec only rejects explicit `snapshot.emit`/`snapshot.inherit` when no effective target tuple exists; implicit auto-emit must be skipped when no target can be resolved, and plans without authored lineage must otherwise keep running.
  - Evidence: The target slug is required at `crates/rhei-cli/src/main_parts/snapshot_runtime_2.rs:32-33`, while the early return for states with no `snapshot.inherit` is later at `crates/rhei-cli/src/main_parts/snapshot_runtime_2.rs:66-73`. The spec says plans without `snapshot.emit` or `snapshot.inherit` are unaffected except for best-effort auto-emits (`docs/functional-spec/rhei-snapshots.spec.md:22-28`) and says missing effective target tuples skip auto-emit but reject explicit snapshot operations (`docs/functional-spec/rhei-snapshots.spec.md:970-976`).
  - Suggested fix: Defer `snapshot_target_slug_or_err` until after the state is known to need an authored preload/override, and make auto-emit paths skip silently when no effective target tuple exists. Keep explicit named emit/inherit rejection in validation or as a runtime `snapshot-requires-target` error.

- Q-emit-log-not-session: Snapshot emission can store the run log and a synthetic session id instead of the native agent session transcript.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/snapshots_1.rs
  - Detail: When no `session_dir` is staged, or when the staged directory has no file with the layout extension, `transcript_source_for_snapshot` falls back to the agent log and emits `transcript.log`. The manifest then records a fabricated `session_id` based on task/state/target/generation rather than the actual agent session id. Downstream preload uses that manifest session id for resume and the stored transcript path for fork, so snapshots may be listed as compatible but cannot actually resume the captured native conversation.
  - Evidence: The fallback to `log_path` is at `crates/rhei-cli/src/main_parts/snapshots_1.rs:276-287`, transcript bytes are copied from that source at `crates/rhei-cli/src/main_parts/snapshots_1.rs:336-352`, and the manifest writes `session_id: format!("{task_id}-{emitting_state}-{target_slug}-g{generation}")` at `crates/rhei-cli/src/main_parts/snapshots_1.rs:355-380`. The spec defines snapshots as completed agent invocation state sufficient for continuation (`docs/functional-spec/rhei-snapshots.spec.md:72-73`), requires copying/renaming the native agent transcript into the generation directory (`docs/functional-spec/rhei-snapshots.spec.md:493-497`), and requires `session_id` to be the session id used by the agent (`docs/functional-spec/rhei-snapshots.spec.md:597`).
  - Suggested fix: Use the declared `SessionLayout` and any session-dir staging to locate the actual native transcript, derive the real session id from the agent/session file or header, and fail named emit with `unsupported-snapshot-session` or skip auto-emit when the native transcript cannot be located. Do not write a log file as a preloadable snapshot transcript.

- Q-poll-self-loop-emits: Polling states emit snapshots on every attempt instead of only on terminal poll exit.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/snapshot_runtime_1.rs
  - Detail: `emit_snapshots_after_agent_exit` only suppresses final, gating, and program states; it has no visibility into whether a polling state selected a self-loop or terminal exit transition. The run loop calls it immediately after the subprocess exits, before poll routing finishes, so both auto and named snapshots are written for poll self-loop attempts. This violates the spec's rule that poll attempts keep the visit open and emit only when the poll succeeds or exhausts.
  - Evidence: The suppression check is limited to `terminal || gating || program.is_some()` at `crates/rhei-cli/src/main_parts/snapshot_runtime_1.rs:43-48`, and the sequential/parallel run paths call emission directly after spawn at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:837-860` and `crates/rhei-cli/src/main_parts/run_agent_mode.rs:1253-1276`. The spec requires auto/named emit suppression on poll self-loop attempts and emission only on terminal exit (`docs/functional-spec/rhei-snapshots.spec.md:856-860`, `docs/functional-spec/rhei-snapshots.spec.md:888-890`).
  - Suggested fix: Select the outgoing transition, including poll self-loop/exhaustion, before snapshot emission and pass that outcome into the emit hook. Skip emission when the selected poll transition loops back to the same state.

- Q-ancestor-select-state-order: `from: ancestor` chooses the nearest ancestor before applying `select.state`.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/snapshot_runtime_1.rs
  - Detail: Runtime ancestor resolution first picks the nearest ancestor with any orchestrator-produced snapshot matching the inherited name, and only then applies `select.state`. If that nearest ancestor has the same snapshot name from a different state while a farther ancestor has the requested selected state, the implementation returns the missing-snapshot fallback instead of skipping the nearer non-matching ancestor.
  - Evidence: The ancestor loop collects records by task id only at `crates/rhei-cli/src/main_parts/snapshot_runtime_1.rs:225-238`, while the `selected_state` filter runs later at `crates/rhei-cli/src/main_parts/snapshot_runtime_1.rs:242`. The spec says the first ancestor-selection phase must choose the nearest ancestor that has at least one emitter matching `inherit.name` and optional `select.state`; ancestors with no such emitter are skipped (`docs/functional-spec/rhei-snapshots.spec.md:248-256`).
  - Suggested fix: Apply `select.state` during the ancestor walk before deciding an ancestor is the nearest match. Keep target, visit, and generation as post-filters after the matching ancestor is selected.
