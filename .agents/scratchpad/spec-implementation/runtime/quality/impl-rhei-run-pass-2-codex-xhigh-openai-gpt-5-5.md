# Quality Review pass 2 - target codex[xhigh]:openai:gpt-5.5

- Q-agent-no-outputs-skips-spawn: Agent states with no `outputs:` are auto-transitioned without spawning the agent.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/run_agent_mode.rs
  - Detail: The run loop filters resolved agent invocations by checking whether their output artifacts already exist before spawning. For a state that declares no `outputs:`, that check succeeds vacuously, so `pending` is empty and the task is routed through callback-only advancement. This skips the agent subprocess entirely even though the spec says the subprocess must still run and exit code alone suffices when no outputs are declared.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:61-66` requires resolving and spawning the state's subprocess before evaluating completion, and `docs/functional-spec/rhei-agents.spec.md:568-572` says a no-output agent state is complete after subprocess exit because the output clause is vacuously true. `run_agent_mode` filters invocations with `!state_outputs_exist_for_resolved_invocation(...)` at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:189-203`; when that list is empty it pushes the task into `callback_tasks` at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:205-207`, so an output-less agent state never reaches the spawn paths.
  - Suggested fix: Treat states with no declared outputs as pending work for resolved agent invocations. Only use pre-existing outputs to skip an invocation when the state actually declares output artifacts for that invocation/resume case.

- Q-model-only-missing-agent-falls-back: Model-driven states without a resolved agent can silently enter callback mode instead of failing.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/run_command.rs
  - Detail: Mode selection uses `resolve_agent_invocations(...).is_empty()` to decide whether agent mode is needed. If a reachable state declares `model` or `all_models` but no state/default/model `default_agent` resolves, `resolve_legacy_agent_with_model` returns `Ok(None)`, `should_use_agent_mode` can return `false`, and the command runs callback-only transitions. The spec requires a missing-agent configuration error for model/target-driven work with no agent transport.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:57` says model/target-driven work must not silently fall back to callback-only transitions when no agent transport resolves. `should_use_agent_mode` only enters agent mode when resolved invocations are non-empty at `crates/rhei-cli/src/main_parts/run_command.rs:64-70`, while `resolve_legacy_agent_with_model` returns `Ok(None)` when no agent is found at `crates/rhei-cli/src/main_parts/agent_resolution.rs:107-123`. Callback mode then applies normal transitions at `crates/rhei-cli/src/main_parts/run_callback_mode.rs:120-170`.
  - Suggested fix: During mode selection and ready-task handling, distinguish "state declares autonomous agent/model work but no agent resolved" from "state declares no autonomous work". For reachable ready model/target states, surface the existing missing-agent error instead of selecting callback mode.

- Q-program-zero-missing-outputs-aborts: Successful program exits with missing required outputs abort through transition validation instead of leaving the task in place.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/run_agent_mode.rs
  - Detail: For a program exit code `0` with a matching `exit_code: 0` transition, the run loop directly calls `execute_system_program_exit_transition`. That transition enforces source `outputs:` and returns an error when outputs are missing; the caller propagates it with `?`. The program spec requires a zero exit with missing required outputs to leave the task in its current state, not to abort the run through a transition error.
  - Evidence: `docs/functional-spec/rhei-programs.spec.md:319-320` says required outputs are checked before committing the exit-code transition and that a zero exit with a missing output leaves the task in place. The program path calls `execute_system_program_exit_transition(...)?` at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:541-553`. That helper sets `skip_source_outputs: exit_code != 0` at `crates/rhei-cli/src/main_parts/system_transitions_1.rs:109-113`, so exit `0` reaches `ensure_state_outputs_exist_for_transition` at `crates/rhei-cli/src/main_parts/system_transitions_2.rs:306-314` and errors instead of being handled as a non-advancing completion failure.
  - Suggested fix: Check required program outputs explicitly after a zero exit and before applying the exit-code transition. If any are missing, log a warning analogous to the agent missing-output path and continue without calling transition execution.

- Q-fanout-split-by-parallel-limit: `all_models` / `all_targets` invocations for one task are sliced by `--parallel` instead of spawned together.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/run_agent_mode.rs
  - Detail: The implementation represents each resolved agent invocation as its own `agent_tasks` entry, then computes the batch size from the number of entries. With `--parallel 1` and one ready task that resolves multiple `all_models` or `all_targets` invocations, only the first invocation is spawned in that pass. The spec says the parallel limit counts tasks, and that every resolved invocation for a scheduled task is spawned together regardless of the state `concurrent` flag.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:61` says up to `--parallel` tasks from the ready set are scheduled, and `docs/functional-spec/rhei-run.spec.md:137` says within-task fanout invocations are still spawned together. `resolve_agent_invocations` expands `all_models` into multiple resolved agents at `crates/rhei-cli/src/main_parts/agent_resolution.rs:220-232`; `run_agent_mode` pushes one `agent_tasks` entry per resolved invocation at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:223-230`, then slices the vector by `max_parallel` at `crates/rhei-cli/src/main_parts/run_agent_mode.rs:645-648`.
  - Suggested fix: Batch by task, not by invocation. Select up to `--parallel` ready task ids, then include all resolved invocations for each selected task in the spawn set.
