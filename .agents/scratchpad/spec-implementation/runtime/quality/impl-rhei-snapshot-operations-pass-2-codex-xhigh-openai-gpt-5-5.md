# Quality Review pass 2 - target codex[xhigh]:openai:gpt-5.5

- Q-staging-manifests-visible: Snapshot readers parse in-progress `g<N>.tmp-*` generations as real cache records.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/snapshots_2.rs
  - Detail: `read_snapshot_records` recursively collects every `manifest.json` under the cache, but the atomic writer creates a `manifest.json` inside `g<N>.tmp-*` before renaming the staging directory to `g<N>`. A concurrent reader, or any stale staging directory left after a crash between manifest write and rename, will fail schema validation because the directory name is not exactly `g<N>`. That makes `snapshot list`, `show`, `gc`, and run-time preload fail on unrelated valid snapshots, which breaks the atomic-write contract by exposing staging state to readers.
  - Evidence: The spec says operator generations use the same atomic-write procedure as other emissions, with next generation allocated via collision-retry, and snapshot operations list/show/gc operate on cache contents (docs/functional-spec/rhei-snapshot-operations.spec.md:170-175, 29-78). The reader descends into every directory and records any `manifest.json` (crates/rhei-cli/src/main_parts/snapshots_2.rs:117-155), while the writer creates `g{generation}.tmp-{nonce}`, writes `manifest.json` there, and only then renames it to `g{generation}` (crates/rhei-cli/src/main_parts/snapshots_1.rs:477-523). Validation rejects non-`g<N>` generation directories (crates/rhei-cli/src/main_parts/snapshots_2.rs:305-312).
  - Suggested fix: Make manifest discovery ignore staging directories, or more narrowly only accept manifests whose parent directory name is exactly `g<N>` under a valid identity path. Optionally clean stale `*.tmp-*` directories under the identity lock before writes.

- Q-run-override-selectors-source-scoped: `--from-snapshot` selectors do not identify a single active run task/fanout invocation.
  - Severity: high
  - File: crates/rhei-cli/src/main_parts/snapshot_runtime_2.rs
  - Detail: The spec requires ambiguous run override contexts to be narrowed to exactly one task and fanout invocation. The implementation calls the override hook once per task/invocation and applies `--task`/`--target` only to the source snapshot record: `--task` is compared with `record.task_id`, and `--target` is passed as a target filter to `resolve_snapshot_ref`. There is no check that the current task id or current resolved target slug matches those selectors, and no preflight error when more than one current invocation would consume the same override. In ancestor or fanout cases, a debug override can therefore be applied to multiple live invocations instead of exiting non-zero with candidates.
  - Evidence: The spec says selectors such as `--task <id>` and `--target <slug>` must make the command identify exactly one task and one fanout invocation (docs/functional-spec/rhei-snapshot-operations.spec.md:194-197). The override hook is invoked for each serial and parallel agent spawn (crates/rhei-cli/src/main_parts/run_agent_mode.rs:779-794, 1150-1163). Inside the hook, `--target` is used only as `resolve_snapshot_ref(..., opts.snapshot_target_selector(), None)`, and `--task` is rejected when it differs from the resolved source snapshot task (crates/rhei-cli/src/main_parts/snapshot_runtime_2.rs:87-106).
  - Suggested fix: Treat the run override selectors as current-run selectors before preload: reject non-matching current tasks/targets, and if `--from-snapshot` would affect multiple active task/target invocations without selectors, abort with candidate invocations. Keep source snapshot disambiguation explicit in the reference or separate source-specific selectors.

- Q-gc-reachability-ignores-source-axis: GC's active-inherit interlock protects snapshots that are not reachable sources.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/snapshots_3.rs
  - Detail: The live-run interlock is supposed to refuse deletion only when the selected generation is reachable from an active `snapshot.inherit` on a non-terminal task. The implementation checks name and selector fields, but it never applies the `snapshot.inherit.from` axis against the active task. As a result, a task in a non-terminal state with the default `from: self` can protect snapshots belonging to unrelated tasks, and `from: ancestor` can protect non-ancestor snapshots if the name/state/target/generation selector happens to match. This causes false GC refusals unless the operator passes `--force`, which is stronger than the specified best-effort interlock.
  - Evidence: The spec frames protection in terms of reachability from the active inherit rule (docs/functional-spec/rhei-snapshot-operations.spec.md:84-92). `snapshot_generation_protected_by_active_inherit` iterates non-terminal tasks and checks `inherit.name`, `select.state`, `select.target`, `select.visit`, and `select.generation`, but has no comparison between `record.task_id` and the active task for `from: self` or `from: ancestor` (crates/rhei-cli/src/main_parts/snapshots_3.rs:315-379).
  - Suggested fix: Reuse the same source-scope predicate used by inherit resolution: for `from: self`, protect only records for the active task and permitted prior visits; for `from: ancestor`, protect only ancestor task ids. Apply that before evaluating generation selectors.

- Q-redactor-default-env-empty: The redactor process does not receive the required default context environment.
  - Severity: medium
  - File: crates/rhei-cli/src/main_parts/snapshots_1.rs
  - Detail: The redactor hook clears the environment and forwards only user-allowlisted `redactor_env` keys. The spec requires a closed environment by default, but not an empty one: it must still include variables needed to locate rhei, the workspace, and the configured settings path. Without those defaults, a compliant redactor cannot reliably discover the workspace/settings context unless every project redundantly allowlists external variables, and success-path redactor metadata is not logged as specified.
  - Evidence: The redactor contract says the program runs in the workspace root and receives a minimal default environment containing variables needed to locate rhei, the workspace, and the configured settings path; it also requires path/status/timeout/truncation/stderr summary logging (docs/functional-spec/rhei-snapshot-operations.spec.md:255-271). The implementation calls `.env_clear()` and then only loops over `snapshot_settings.redactor_env`, with no `RHEI_*` or settings-path defaults and no success-path logging (crates/rhei-cli/src/main_parts/snapshots_1.rs:319-333, 392-411).
  - Suggested fix: Define and document the minimal default redactor environment, e.g. rhei executable path, workspace root, and project/global settings paths, then add those variables after `env_clear()` before applying the allowlist. Emit redactor diagnostics on success and failure through the run log path used by snapshot emission.
