# Quality Review pass 2 - target codex[xhigh]:openai:gpt-5.5

- Q-defaults-only-agent-mode-skipped: `rhei run` stays in callback mode when agent execution is configured only by CLI or settings defaults
  - Severity: high
  - File: crates/rhei-cli/src/main.rs
  - Detail: The dispatcher enters `run_agent_mode` only when a non-terminal, non-gating state explicitly declares `agent`, `model`, `target`, `all_models`, `all_targets`, or `program`. It ignores `--agent`, `--model`, `defaults.agent`, `defaults.model`, and `models.<id>.default_agent`, even though those are part of the specified resolution order. A plan that relies on global/project defaults, or a user invoking `rhei run --agent codex`, silently runs the legacy callback loop instead of spawning an autonomous agent.
  - Evidence: The spec makes `--agent`, `--model`, settings defaults, and model defaults valid resolution inputs for autonomous agent launch (`docs/functional-spec/rhei-agents.spec.md:270-306`) and says `--no-agent` is the flag that disables agent spawning (`docs/functional-spec/rhei-agents.spec.md:652-657`). The mode switch only calls `state_declares_enabled_autonomous_execution(def, &opts)` (`crates/rhei-cli/src/main.rs:9872-9884`), and that predicate checks only per-state fields, not CLI overrides or merged settings defaults (`crates/rhei-cli/src/main.rs:12143-12154`).
  - Suggested fix: Decide agent mode from the effective run configuration, not just per-state declarations: when `--no-agent` is false and an agent/model can resolve from CLI/settings/model defaults for a runnable state, enter `run_agent_mode` and let the existing resolver produce the configured agent or the specified "no agent configured" error.

- Q-missing-outputs-respawn-loop: Successful agents with missing required outputs are silently rescheduled instead of warning and stopping that attempt
  - Severity: high
  - File: crates/rhei-cli/src/main.rs
  - Detail: After an agent exits 0, the run loop calls `task_has_pending_agent_invocations`; if any required output is still absent, it immediately `continue`s before logging the required missing-output warning or ending the pass. For a single-invocation state, this makes a well-behaved agent that exits 0 without producing an output get spawned again on the next pass, potentially in a tight paid-work loop, and the operator never sees the spec-required diagnostic.
  - Evidence: The spec requires an exit-0 agent to verify outputs, leave the task in its current state, and log `warning: agent exited 0 but required outputs are missing...` when any required output is missing (`docs/functional-spec/rhei-agents.spec.md:598-604`). The sequential path skips directly on `pending_more` (`crates/rhei-cli/src/main.rs:10701-10720`) before the warning path (`crates/rhei-cli/src/main.rs:10739-10748`), and the parallel result path has the same early `continue` (`crates/rhei-cli/src/main.rs:11027-11050`).
  - Suggested fix: Distinguish "other fanout invocations still need to run" from "this completed invocation did not satisfy required outputs"; for the latter, emit the missing-output warning, do not auto-advance, and do not immediately reschedule the same invocation in the same run loop.

- Q-tooling-availability-not-checked: Required MCP availability is not actually probed before spawn
  - Severity: high
  - File: crates/rhei-cli/src/main.rs
  - Detail: The availability gate only checks whether an MCP entry has a registry/inline definition and whether the resolved agent exposes an MCP flag. It does not launch command-based MCP servers, connect to URL-based servers, perform MCP handshakes, honor `startup_timeout`, or verify that a resolved MCP process remains alive before the agent spawn. A required MCP server with a missing binary or unreachable URL will still be treated as available and passed to the agent, instead of taking the `mcp_unavailable` path.
  - Evidence: The spec defines MCP availability as command launch plus handshake, or URL connection plus handshake, within `startup_timeout`; required failures must prevent agent spawn or fire `mcp_unavailable` (`docs/functional-spec/rhei-agents.spec.md:736-763`). The implementation documents that availability is currently "registry resolution only" and that actual MCP handshake checks are still future work (`crates/rhei-cli/src/main.rs:6765-6770`); `gate_tooling_for_agent` only tests `entry.definition.is_none()` and agent flag support (`crates/rhei-cli/src/main.rs:7997-8028`).
  - Suggested fix: Add a pre-spawn MCP availability phase that probes each resolved MCP profile according to its transport and `startup_timeout`, marks optional failures unavailable, and routes required failures through `mcp_unavailable` before constructing the agent command.

- Q-optional-tooling-disappears: Optional unavailable tooling is dropped before env vars, templates, and logs can report it
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: Optional unavailable MCP/skill entries are removed from `gate.tooling` entirely. The prompt render context, agent environment, and log header receive only that filtered set, so the agent never gets `RHEI_MCP_<ID>_AVAILABLE=false` / `RHEI_SKILL_<ID>_AVAILABLE=false`, `{mcp.<id>.available}` / `{skill.<id>.available}` cannot reflect the declared unavailable entry, and the log cannot render the `?` diagnostic suffix.
  - Evidence: The spec says optional failures warn but spawn with remaining tooling, while prompt template variables and env vars reflect availability (`docs/functional-spec/rhei-agents.spec.md:764-779`); the log format requires optional failed entries to be recorded with `?` (`docs/functional-spec/rhei-agents.spec.md:974-977`). The gate records a warning for optional failures but does not push the failed entry into `result.tooling` (`crates/rhei-cli/src/main.rs:8013-8028`, `crates/rhei-cli/src/main.rs:8040-8055`), and env/log emission only iterates entries present in the filtered `ResolvedTooling` (`crates/rhei-cli/src/main.rs:8167-8181`, `crates/rhei-cli/src/main.rs:8370-8380`).
  - Suggested fix: Preserve declared-but-unavailable optional entries in the tooling object passed to prompt/env/log rendering, while omitting only their CLI attachment flags from the spawned command.

- Q-target-selector-overridden-by-registry: `target` / `all_targets` model fields can be overwritten by a same-named model profile
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: Target selectors are supposed to bypass normal model resolution for the encoded provider/model fields. The resolver nevertheless looks up `settings.models[target.model]` and replaces the target's concrete model name with `models.<id>.model` when a registry entry happens to share that name. This can spawn a different provider model than the selector literally requested, while logs/env still show the selector value as the model id.
  - Evidence: The spec says that for `all_targets`, resolution is bypassed and "provider, and model name come from the selector itself" (`docs/functional-spec/rhei-agents.spec.md:309-318`). `resolve_target_agent` consults `settings.models.get(target.model.as_str())`, then sets `model_name` from `model_profile.model` instead of always using `target.model` (`crates/rhei-cli/src/main.rs:7070-7093`).
  - Suggested fix: For `target` / `all_targets`, use the selector's provider/model as the effective provider/model name for prompt/env/command; only consult a matching registry entry for non-identity metadata such as model-agent timeout bindings if that compatibility is still desired.

- Q-mode-default-order-lost: The registry default mode is lexicographic, not the first declared mode
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: When no CLI/state/default `agent_mode` is set, the implementation chooses `profile.modes.keys().next()`. Because modes are stored in a `BTreeMap`, this chooses the lexicographically smallest mode name, not the first mode declared in settings. A custom profile that declares `"yolo"` first and `"safe"` second will default to `"safe"`, violating the transport contract and potentially changing the expected permission/sandbox posture.
  - Evidence: The spec defines the registry default as "the first declared mode in the agent entry's `modes` map" (`docs/functional-spec/rhei-agents.spec.md:320-330`). `CustomAgentProfile.modes` is a `BTreeMap` (`crates/rhei-validator/src/lib.rs:192-196`), and legacy resolution uses `profile.modes.keys().next().cloned()` (`crates/rhei-cli/src/main.rs:7180-7182`).
  - Suggested fix: Preserve declaration order for `modes` during settings deserialization, or make the default explicit in the agent profile schema and require users to set `agent_mode` when order cannot be represented.

- Q-tooling-unknown-id-not-validation: Unknown MCP/skill ids are treated as spawn-time unavailability instead of configuration errors
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: A state or defaults entry that references an MCP server or skill id missing from the merged registry is allowed through validation and later becomes `definition = None`. That misclassifies a static configuration typo as a runtime availability failure, so optional typos can be silently dropped and required typos can take `*_unavailable` transitions instead of failing validation/settings load as specified.
  - Evidence: The spec classifies "Id referenced in a state but not in the merged registry and not an inline entry" as a `rhei validate` / settings-load hard error (`docs/functional-spec/rhei-agents.spec.md:733-736`). The CLI settings validator self-validates registries and state agents/targets but does not validate state/default MCP or skill ids against the merged registries (`crates/rhei-cli/src/main.rs:6612-6674`), while `resolve_mcp_entry` returns `definition: None` on registry miss (`crates/rhei-cli/src/main.rs:6912-6921`) and an explicit test locks in "unknown id resolves to unavailable" (`crates/rhei-cli/src/main.rs:15389-15404`).
  - Suggested fix: During validation after settings merge, check every id-only MCP/skill entry from `defaults` and each state against the merged registry; reserve spawn-time availability handling for registry/inline entries whose concrete launch/path checks fail.
