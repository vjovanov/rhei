# Quality Review pass 1 - target codex[xhigh]:openai:gpt-5.5

- Q-program-timeout-misroutes-nonzero: Program failures are treated as timeouts whenever a timeout is configured.
  - Severity: high
  - File: crates/rhei-cli/src/main.rs
  - Detail: `spawn_and_wait_program` returns only `ExitStatus`, so the run loop cannot tell whether the timeout watchdog actually fired. As a result, any non-zero program exit with `program_timeout`/`--program-timeout` configured is reported as `TimedOut` and attempts a timeout transition before normal `exit_code` routing. A fast deterministic failure can therefore be routed to a timeout state, and a real timeout with no timeout rule can still fall through to exit-code handling instead of remaining in place with the timeout warning required by the spec.
  - Evidence: `docs/functional-spec/rhei-programs.spec.md:152` says program exit-code evaluation uses the program's actual exit after it has not already advanced the task, while `docs/functional-spec/rhei-programs.spec.md:199` limits timeout-transition behavior to actual timeout handling. `spawn_and_wait_program` returns `MietteResult<std::process::ExitStatus>` with no timed-out bit (`crates/rhei-cli/src/main.rs:8711`, `crates/rhei-cli/src/main.rs:8753`). The caller then classifies any non-success status with any configured timeout as `TaskOutcome::TimedOut` (`crates/rhei-cli/src/main.rs:10383`) and calls `fire_timeout_transition` for `!status.success() && resolved.timeout_secs.is_some()` (`crates/rhei-cli/src/main.rs:10430`) before evaluating `find_program_exit_transition`.
  - Suggested fix: Return a program spawn outcome that carries `status`, `timed_out`, and `timeout_secs`, matching the agent path, and route through timeout transitions only when `timed_out == true`; otherwise go directly to exit-code evaluation.

- Q-failure-transitions-require-success-outputs: Timeout and non-zero failure transitions are blocked by required output artifacts.
  - Severity: high
  - File: crates/rhei-cli/src/main.rs
  - Detail: All transition application goes through `execute_transition_with_origin`, which unconditionally checks the source state's required `outputs:` before changing state. That is correct for successful completion transitions, but not for failure routes: a timed-out or non-zero subprocess often has not produced success outputs, and the specs require failure/timeout routing to skip the artifact check. In practice, a timeout transition or program `exit_code: nonzero` transition can fail with "Missing required output artifact" and leave the task in the work state.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:69` requires failed completion to route through the state's error or timeout transition, and `docs/functional-spec/rhei-agents.spec.md:595` says the artifact check is skipped on non-zero exit. Program exit-code transitions call `execute_transition` at `crates/rhei-cli/src/main.rs:10491`, and timeout transitions call `execute_system_timeout_transition` at `crates/rhei-cli/src/main.rs:11847`; both share `execute_transition_with_origin`, which unconditionally runs `ensure_state_outputs_exist_for_transition` for the source state at `crates/rhei-cli/src/main.rs:5655`.
  - Suggested fix: Thread a transition reason/origin through transition execution and skip source-output validation for timeout, tooling-unavailable, and non-zero/failed completion routes while preserving it for success/normal transitions.

- Q-poll-max-attempts-off-by-one: Poll `max_attempts` allows one extra subprocess attempt.
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: `stateVisits` is specified as the poll attempt counter, and once it reaches `poll.max_attempts` the engine must refuse self-loops. The implementation increments `stateVisits` only after a self-loop transition has already been selected and persisted for the next pass. Selection checks the old count with `current < max_attempts`, so `max_attempts: 1` still selects a self-loop after the first subprocess attempt, sleeps, and runs a second subprocess before exhaustion logic can apply.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:118` defines `stateVisits.<state-name>` as the attempt counter, and `docs/functional-spec/rhei-run.spec.md:124` requires self-loops to be refused once it reaches `poll.max_attempts`. The self-loop applicability check uses the current persisted count (`crates/rhei-cli/src/main.rs:4104`), but the count is incremented later in `set_poll_next_attempt_metadata` (`crates/rhei-cli/src/main.rs:4364`) via `record_poll_self_loop_if_needed` (`crates/rhei-cli/src/main.rs:4664`), after transition selection.
  - Suggested fix: Count the current subprocess attempt before self-loop applicability is evaluated, or compare `current + 1` against `poll.max_attempts` when deciding whether a self-loop may be selected.

- Q-program-states-ignore-concurrent-flag: Program states bypass the concurrent-state scheduling rule.
  - Severity: medium
  - File: crates/rhei-cli/src/main.rs
  - Detail: The spec applies `concurrent: false` to all ready tasks in a state, but the implementation only filters agent tasks. Ready program tasks are pushed into `program_tasks` without a per-state claimant and then every entry is executed in the same pass. Multiple tasks in the same default non-concurrent program state can therefore run in one pass instead of leaving the extras ready for later passes.
  - Evidence: `docs/functional-spec/rhei-run.spec.md:132` says the `concurrent` flag determines whether multiple ready tasks in the same state may be scheduled together, and `docs/functional-spec/rhei-run.spec.md:135` says `concurrent: false` allows at most one ready task per state per pass. Program states are collected unfiltered at `crates/rhei-cli/src/main.rs:10141` and iterated at `crates/rhei-cli/src/main.rs:10341`; the state-claimant/deferred filtering is applied only afterward to `agent_tasks` at `crates/rhei-cli/src/main.rs:10555`.
  - Suggested fix: Apply the non-concurrent state claimant filter before splitting ready work by execution kind, or add the same filter/defer behavior to `program_tasks`.
