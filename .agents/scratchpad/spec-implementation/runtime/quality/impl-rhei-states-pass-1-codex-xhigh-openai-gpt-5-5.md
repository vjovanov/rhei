# Quality Review pass 1 - target codex[xhigh]:openai:gpt-5.5

- Q-poll-max-attempts-off-by-one: Poll self-loops stop one attempt early for `max_attempts > 2`
  - Severity: high
  - File: crates/rhei-cli/src/cli/metadata_conditions.rs
  - Detail: `loop_reentry_allowed` decides poll self-loop eligibility from the persisted `stateVisits` value and requires `current + 1 < poll.max_attempts`. After the first self-loop, `record_poll_self_loop_if_needed` persists the next current attempt count. For `max_attempts: 3`, the second attempt has `stateVisits == 2`; the self-loop is rejected because `2 + 1 < 3` is false, but the exhaustion transition condition `pollAttempts >= pollMaxAttempts` is also false (`2 >= 3`). The task then stalls before the allowed third attempt and before exhaustion handling can fire. The same pattern prevents the final allowed retry for any poll cap greater than two.
  - Evidence: The spec says the counter starts at `1`, increments on every self-loop re-entry, and self-loops are refused only when `stateVisits.<state-name> >= poll.max_attempts`; exhaustion then selects a non-self-loop or fails clearly. §FS-rhei-states. Code instead uses `task_visit_count(...).saturating_add(1) < poll.max_attempts` in `loop_reentry_allowed` (`crates/rhei-cli/src/cli/metadata_conditions.rs:252-256`) while `record_poll_self_loop_if_needed` stores `current_state_visit_count(...) + 1` as the next attempt count (`crates/rhei-cli/src/cli/transition_context.rs:269-279`).
  - Suggested fix: Base poll self-loop eligibility on the current attempt index and allow retry while `current_attempt < poll.max_attempts`; use `current_state_visit_count(...)` (or equivalent) so first entry is counted as attempt `1`, and add a regression for `max_attempts: 3` proving the third attempt runs and exhaustion is reachable.

- Q-program-poll-bypasses-normal-transition-selection: Program-backed polling does not evaluate normal poll transitions after a successful program exit
  - Severity: medium
  - File: crates/rhei-cli/src/cli/programs.rs
  - Detail: The program path only calls `find_program_exit_transition`, which considers transitions that declare `exit_code`; for exit code `0` it returns `None` when no exact numeric `exit_code` transition exists. That bypasses the normal transition matcher used by agent/callback paths, so a polling program with the spec-shaped self-loop (`from: wait, to: wait`) and exhaustion condition (`condition: pollAttempts >= pollMaxAttempts`) but no `exit_code` never records `pollNextAttemptAt` and never reaches poll exhaustion. Authors can satisfy the states validator with a self-loop that `rhei run` will not actually use for program polling.
  - Evidence: The states spec says each poll attempt's `agent` or `program` runs once and the engine evaluates transitions normally; any selected self-loop means retry. §FS-rhei-states. The run spec also selects the first transition whose `condition` / `exit_code` matches. §FS-rhei-run. In the program execution path, `run_agent_mode` calls `find_program_exit_transition` after the subprocess exits (`crates/rhei-cli/src/cli/run_agent_mode.rs:612-618`), and that helper filters to numeric/sequence `exit_code` transitions, returns `None` immediately on exit code `0` if no such rule matched, and never considers plain or condition-only poll transitions (`crates/rhei-cli/src/cli/programs.rs:241-291`).
  - Suggested fix: For successful program exits from poll states, run the same normal transition selection used by agent/callback completion before falling back to program-specific exit-code warnings; alternatively require and validate `exit_code` on poll program self-loop/exhaustion transitions, but that would need a spec change.
